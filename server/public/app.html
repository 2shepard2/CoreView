Profiles

CREATE NEW

Profiles define persistent screen content. Assign a profile to a screen to set what it normally shows, then use rules for temporary overrides when events, state changes, or schedules require it.

Create and manage reusable widgets in the Widgets tab, then reference up to 4 here by slot.

Widgets

CREATE NEW

Widgets are reusable building blocks for custom profiles and widget override rules. Define them once, then reuse by ID everywhere.

Assign widgets to custom profile slots in the Profiles tab, then use Rules or Manual Control to swap specific slots dynamically.

Manual Override

Apply a saved profile, theme, banner, ticker, or a live camera feed to the selected target. Leave duration blank for a persistent manual change on runtime-backed overrides. Camera feeds remain time-limited overlays.

This manual change will persist until the screen is changed by another rule, a new screen assignment, or another manual action.
Use saved-object overrides for theme, banner, ticker, and profile. Use Manual Rule Trigger to execute a saved non-overlay rule immediately. Use Live Camera Feed for a direct manual camera overlay on the selected target.

Each card shows the effective state of a screen right now, including temporary rule-driven overrides. Use this to confirm what a screen is actually running instead of what it is assigned by default.

Use this as the first checkpoint before changing any integration settings. It reflects current connectivity and health across the services CoreView depends on.

Backups

Export and restore the CoreView configuration database. Nightly backups are written automatically, and manual export gives you a portable JSON snapshot for rollback or migration.

Danger zone: import is a full replacement restore. It overwrites the current CoreView configuration and does not merge changes.

How CoreView Is Organized

CoreView is section-based. Top navigation selects a section, section navigation selects a tool, and each tool works on saved configuration objects.

Views
Runtime abstraction layer. Views are what screens and URL consumers actually render.
Design
Profiles, widgets, and themes define reusable content and presentation primitives.
Automation
Rules, banners, tickers, and manual overrides apply temporary runtime changes.
Targets
Screens and groups define where runtime changes are delivered.
System
Operational state, integration setup, security-sensitive settings, and backup/restore.

Recommended Operator Flow

Use this sequence to keep permanent configuration and temporary runtime behavior clearly separated.

1. In System -> Setup, confirm integrations and credentials are healthy.
2. In Design, create widgets, profiles, and themes.
3. In Views, bind a profile (and optional theme) into a publishable runtime unit.
4. In Targets, register screens and assign their default view; use groups for reusable targeting.
5. In Automation, create rules and reusable banners/tickers for temporary runtime behavior.
6. Use Manual Control for ad-hoc operator actions, not baseline configuration.
7. Use System -> Status to verify effective runtime and last applied rule outcomes.

Runtime Model (What Actually Wins)

CoreView computes effective runtime by layering defaults and temporary actions. This is the main mental model for troubleshooting.

Base runtime comes from the screen's assigned View.
View references a Profile and optionally a Theme.
Rules and manual overrides can temporarily replace parts of runtime.
Durations/cooldowns govern automatic restoration behavior.
Status shows effective runtime, not just configured defaults.

Operations, Security, And Recovery

Operational safety in CoreView comes from scoped credentials, observability, and recoverable config snapshots.

System -> Setup manages Home Assistant, MQTT, Frigate, Immich, and event webhook settings.
Backups run nightly at 2:00 AM server time; export/import is full replacement, not merge.
Secret integration values are stored encrypted for this instance.
Treat webhook and API credentials as sensitive and rotate on exposure.
